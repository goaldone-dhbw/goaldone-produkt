# Phase 6.1: Fix Database Schema Validation Errors

**Status:** Ready for planning  
**Priority:** Critical (blocks application startup)  
**Estimated Effort:** 8-12 hours

---

## Goal

Fix Hibernate schema validation error caused by missing `acceptance_reason` column in the `invitations` table. Ensure database migrations are properly applied and the application can start with schema validation enabled.

---

## Problem Statement

**Error:** `Schema validation: missing column [acceptance_reason] in table [invitations]`

**Root Cause:** The invitations entity has a field `acceptanceReason` mapped in the JPA model, but the corresponding database column is missing from the schema.

**Impact:**
- Application fails to start with Hibernate schema validation enabled
- Blocks all development and testing workflows
- Prevents local and remote deployment

---

## Scope

**In Scope:**
- Identify all missing columns in the invitations table
- Create Liquibase migration to add missing columns with appropriate constraints
- Verify schema matches the current JPA entity definitions
- Test migration on clean and existing databases
- Enable schema validation on startup

**Out of Scope:**
- Refactoring invitation entity structure
- Data transformation or backfill logic (beyond migration requirements)
- Performance optimization of invitations table

---

## Tasks

### 1. Audit JPA Entities and Database Schema

**Files to Review:**
- `src/main/java/de/goaldone/authservice/model/entity/Invitation.java`
- `src/main/resources/db/changelog/invitations_*.xml` (all migration files)
- Current database schema via `INFORMATION_SCHEMA`

**Actions:**
- Extract all JPA column definitions from Invitation entity
- Compare with current database schema
- Document all discrepancies and missing columns
- Note any columns that exist in DB but not in entity (consider DROP vs. migration)

**Success Criteria:**
- Complete audit report with all missing/extra columns identified
- Clear mapping between entity fields and database columns

### 2. Create Liquibase Migration for Missing Columns

**File:** `src/main/resources/db/changelog/invitations_add_missing_columns_<timestamp>.xml`

**Changes:**
- Add `acceptance_reason` column (VARCHAR/TEXT based on entity definition)
- Verify column data type, NOT NULL constraints, and default values match entity
- Include rollback statements for each addColumn change
- Use appropriate Liquibase tags for database compatibility

**Migration Details:**
- Column name: `acceptance_reason`
- Type: VARCHAR(500) or TEXT (check entity definition for length)
- Nullable: Verify from `@Column(nullable = ...)` annotation
- Default: NULL or specific default if required

**Success Criteria:**
- Migration file created with proper Liquibase formatting
- All missing columns addressed
- Rollback path defined
- Ready for testing

### 3. Test Migration on Local Database

**Test Scenarios:**
- **Fresh DB:** Apply migrations from scratch, verify schema is complete
- **Existing DB:** Apply new migration to current schema, verify no conflicts
- **Rollback:** Rollback migration and re-apply, verify idempotency

**Database Setup:**
- Create test database: `auth_service_schema_test`
- Run migrations: `mvn liquibase:update`
- Verify with: `SHOW COLUMNS FROM invitations;` (or equivalent)

**Success Criteria:**
- All migrations apply successfully
- No duplicate or conflicting changes
- Schema matches JPA entity structure
- Rollback works without data loss

### 4. Test Application Startup with Schema Validation

**Configuration:** `application.yaml`
```yaml

  jpa:
```

**Test Process:**

**Success Criteria:**
- Application starts successfully
- No schema validation exceptions

- No Hibernate warnings or errors
