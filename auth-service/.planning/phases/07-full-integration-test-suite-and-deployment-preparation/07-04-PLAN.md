   ```yaml
   name: Build and Push Docker Image

   on:
     push:
       branches: [ main ]
       tags: [ 'v*' ]
     workflow_dispatch:  # Allow manual trigger from Actions tab

   jobs:
     build:
       runs-on: ubuntu-latest
       
       permissions:
         contents: read
         packages: write
       
       steps:
       - name: Checkout code
         uses: actions/checkout@v4

       - name: Set up Docker Buildx
         uses: docker/setup-buildx-action@v2

       - name: Log in to Docker Hub
         uses: docker/login-action@v2
         with:
           username: ${{ secrets.DOCKERHUB_USERNAME }}
           password: ${{ secrets.DOCKERHUB_TOKEN }}
         if: github.event_name != 'pull_request'

       - name: Log in to GitHub Container Registry
         uses: docker/login-action@v2
         with:
           registry: ghcr.io
           username: ${{ github.actor }}
           password: ${{ secrets.GITHUB_TOKEN }}
         if: github.event_name != 'pull_request'

       - name: Extract metadata
         id: meta
         uses: docker/metadata-action@v5
         with:
           images: |
             docker.io/${{ secrets.DOCKERHUB_USERNAME }}/auth-service
             ghcr.io/${{ github.repository }}
           tags: |
             type=ref,event=branch
             type=semver,pattern={{version}}
             type=semver,pattern={{major}}.{{minor}}
             type=sha,prefix={{branch}}-
             type=raw,value=latest,enable={{is_default_branch}}

       - name: Build and push Docker image
         uses: docker/build-push-action@v5
         with:
           context: .
           push: ${{ github.event_name != 'pull_request' }}
           tags: ${{ steps.meta.outputs.tags }}
           labels: ${{ steps.meta.outputs.labels }}
           cache-from: type=registry,ref=docker.io/${{ secrets.DOCKERHUB_USERNAME }}/auth-service:buildcache
           cache-to: type=registry,ref=docker.io/${{ secrets.DOCKERHUB_USERNAME }}/auth-service:buildcache,mode=max

       - name: Scan Docker image with Trivy
         uses: aquasec/trivy-action@master
         with:
           image-ref: ${{ steps.meta.outputs.tags }}
           format: 'sarif'
           output: 'trivy-results.sarif'
         continue-on-error: true
       
       - name: Upload Trivy results to GitHub Security
         uses: github/codeql-action/upload-sarif@v2
         with:
           sarif_file: 'trivy-results.sarif'
         if: always()

       - name: Run Snyk Docker image scan
         uses: snyk/actions/docker@master
         env:
           SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
         with:
           image: ghcr.io/${{ github.repository }}:${{ github.sha }}
           args: --severity-threshold=high
         continue-on-error: true
   ```
3. Verify YAML syntax
4. Save the file
</action>

   ## Coverage Enforcement

   ### Coverage Threshold

   The build-and-test workflow enforces a **70% coverage minimum**:

   ```yaml
   - name: Check Coverage Threshold (70%)
     run: |
       COVERAGE=$(grep -oP 'TOTAL.*?<td class="bar"><img src=".*?" width="\K[0-9]+' target/site/jacoco/index.html)
       if [ "$COVERAGE" -lt 70 ]; then
         echo "Coverage $COVERAGE% is below 70% threshold"
         exit 1
       fi
   ```

   **Failure Behavior:**
   - If coverage < 70%, job fails
   - PR cannot be merged until coverage meets threshold
   - Developers must add tests to increase coverage

   ### Improving Coverage

   1. **Identify uncovered code:**
      ```bash
      open target/site/jacoco/index.html  # View coverage report
      ```

   2. **Add tests for uncovered lines** (see `.planning/phases/07-full-integration-test-suite-and-deployment-preparation/07-02-PLAN.md`)

   3. **Run locally to verify:**
      ```bash
      mvn clean test verify
      open target/site/jacoco/index.html
      ```

   4. **Push changes and let CI verify:**
      Coverage report is automatically generated and commented on PR

   ### Coverage Report Details

   JaCoCo generates HTML report at: `target/site/jacoco/index.html`

   **Report includes:**
   - Overall coverage percentage
   - Coverage by package, class, method
   - Line-by-line coverage visualization (green/red)
   - Complexity metrics
   - Branch coverage

   ## Docker Image Versioning

   ### Semantic Versioning

   Docker images follow semantic versioning: `MAJOR.MINOR.PATCH`

   **Examples:**
   - `1.0.0` — Initial release
   - `1.0.1` — Bug fix
   - `1.1.0` — New feature
   - `2.0.0` — Breaking change

   ### Tagging Strategy

   Git tag → Docker image tags:

   ```
   git tag v1.2.0 && git push origin v1.2.0
   ```

   Produces Docker images:
   - `docker.io/username/auth-service:1.2.0` (exact version)
   - `docker.io/username/auth-service:1.2` (minor version)
   - `docker.io/username/auth-service:v1.2.0` (v-prefixed)

   **Branch-based tags:**
   ```
   git push origin main
   ```

   Produces Docker images:
   - `docker.io/username/auth-service:latest` (main branch only)
   - `docker.io/username/auth-service:main-<sha>` (git commit SHA)

   ### Deployment Best Practices

   **Production:** Always deploy using explicit version tags
   ```bash
   docker pull docker.io/username/auth-service:1.2.0
   ```

   **Staging:** Can use `latest` for quick iterations
   ```bash
   docker pull docker.io/username/auth-service:latest
   ```

   **Development:** Use branch-specific images
   ```bash
   docker pull docker.io/username/auth-service:develop-abc1234
   ```

   ## Troubleshooting

   ### Tests Failing in CI

   **Problem:** Tests pass locally but fail in CI

   **Solutions:**
   1. **Check test logs:** Actions tab → workflow run → "Run Unit Tests" step
   2. **Service containers down:** PostgreSQL/Redis service may have failed to start
      - Check service health logs
      - Increase health check timeouts if needed
   3. **Database state:** Integration tests may be leaving data behind
      - Ensure all tests use `@Transactional` with rollback
      - Check for test isolation issues
   4. **Timing issues:** Flaky tests due to race conditions
      - Increase timeouts in assertions
      - Use proper synchronization (e.g., `WebDriverWait` for Selenium)

   **Debug locally:**
   ```bash
   # Replicate CI environment
   docker run -d --name postgres -e POSTGRES_PASSWORD=test postgres:16-alpine
   docker run -d --name redis redis:7-alpine
   mvn test -Dspring.datasource.url=jdbc:postgresql://localhost:5432/postgres
   ```

   ### Coverage Threshold Not Met

   **Problem:** PR fails because coverage < 70%

   **Solutions:**
   1. **View coverage report:**
      - Download artifact from Actions tab
      - Open `target/site/jacoco/index.html` in browser
   2. **Identify uncovered code:**
      - Look for red/yellow lines in HTML report
      - Focus on critical business logic first
   3. **Add tests:** Create integration tests for uncovered flows
   4. **Re-run workflow:** Push new commits with tests
      - Coverage report updates automatically

   **Quick check:**
   ```bash
   mvn clean test verify
   open target/site/jacoco/index.html
   ```

   ### Docker Build Fails

   **Problem:** Docker build step fails

   **Solutions:**
   1. **Check Dockerfile syntax:**
      ```bash
      docker build --dry-run .
      ```
   2. **Maven build failed:** Check "Build with Maven" step logs
      - Verify dependencies resolve
      - Check Java version (should be 21)
   3. **Out of memory:** Docker builder ran out of memory
      - Increase Docker memory in runner settings
      - Try building locally first

   **Debug locally:**
   ```bash
   docker build -t auth-service:test . --progress=plain
   ```

   ### Image Push to Registry Fails

   **Problem:** "docker login failed" error

   **Solutions:**
   1. **Check secrets:** Verify GitHub Secrets are set
      - Repository Settings → Secrets and variables → Actions
      - Confirm `DOCKERHUB_USERNAME` and `DOCKERHUB_TOKEN` exist
   2. **Token expired:** Docker Hub token may have expired
      - Generate new token: [Docker Hub Security Settings](https://hub.docker.com/settings/security)
      - Update GitHub Secret
   3. **Registry offline:** Check registry status
      - Try push manually: `docker login -u username` then `docker push ...`

   **Manual push:**
   ```bash
   docker login -u $DOCKERHUB_USERNAME -p $DOCKERHUB_TOKEN
   docker push docker.io/username/auth-service:1.0.0
   ```

   ### Snyk Scan Fails

   **Problem:** Snyk security scan reports vulnerabilities

   **Solutions:**
   1. **Check Snyk report:** Workflow logs show vulnerability details
   2. **Update dependencies:** Many CVEs fixed by upgrading:
      ```bash
      mvn dependency:tree | grep vulnerability
      # Update pom.xml with newer versions
      ```
   3. **Suppress false positives:** Snyk allows suppressing non-applicable vulns
      - Instructions: [Snyk Documentation](https://docs.snyk.io/)
   4. **Monitor Snyk dashboard:** [snyk.io/org/your-org](https://snyk.io)

   **Continue on error:** If Snyk scan doesn't block merge, non-critical vulns won't fail pipeline

## Notes

**GitHub Actions Workflow Design:**
- Two separate workflows: build-and-test (on every push/PR) and docker-build-push (on main/tags only)
- Build & Test runs fast (10-15 min) with Maven caching and parallel execution
- Docker Build runs only on main branch to avoid registry bloat
- Coverage threshold enforced: PRs cannot merge with < 70% coverage
- Security scanning (Snyk, Trivy) runs but doesn't block merge (non-critical vulnerabilities allowed)

**Docker Image Strategy:**
- Multi-stage build separates compilation from runtime, reducing final image size
- Images tagged with semantic versioning (v1.2.3 → 1.2.3, 1.2, latest)
- Both registries updated simultaneously (Docker Hub + GHCR)
- Build cache preserved in registry for faster subsequent builds

**CI/CD Integration:**
- All configuration via environment variables (no secrets in code)
- GitHub Secrets store credentials securely (never displayed in logs)
- Workflow logs available in Actions tab for debugging
- Artifacts (test results, coverage) stored for 30 days for audit trail

**Next Steps (Phase 8):** Production logging with SLF4J, metrics collection, distributed tracing, and deployment to cloud platforms (Kubernetes, AWS, Azure).
