2. Populate with the following multi-stage Dockerfile:
   ```dockerfile
   # ==============================================================================
   # Stage 1: Builder - Compile application
   # ==============================================================================
   FROM eclipse-temurin:21-jdk-alpine AS builder

   WORKDIR /build

   # Copy Maven wrapper and POM
   COPY mvnw .
   COPY .mvn .mvn
   COPY pom.xml .

   # Cache Maven dependencies (separate Docker layer for faster rebuilds)
   RUN chmod +x ./mvnw && \
       ./mvnw dependency:go-offline -B -DskipTests=true

   # Copy source code
   COPY src src

   # Build application JAR (skip tests for faster builds)
   RUN ./mvnw clean package -B -DskipTests=true -Dmaven.compiler.fork=false

   # ==============================================================================
   # Stage 2: Runtime - Run application
   # ==============================================================================
   FROM eclipse-temurin:21-jre-alpine

   # Create non-root user for security (prevents privilege escalation)
   RUN addgroup -S appgroup && adduser -S appuser -G appgroup

   WORKDIR /app

   # Copy compiled JAR from builder stage
   COPY --from=builder --chown=appuser:appgroup /build/target/auth-service-*.jar app.jar

   # Switch to non-root user
   USER appuser

   # JVM options for containerized environments
   # - UseG1GC: Garbage collector optimized for low-latency, predictable pause times
   # - MaxGCPauseMillis: Target pause time for GC
   # - UseStringDeduplication: Reduce memory footprint
   # - ParallelRefProcEnabled: Improve concurrent GC performance
   # - Xms/Xmx: Explicit heap size (can be overridden via environment)
   ENV JAVA_OPTS="-XX:+UseG1GC -XX:MaxGCPauseMillis=200 -XX:+UseStringDeduplication -XX:+ParallelRefProcEnabled -Xms512m -Xmx1024m"

   # Health check for container orchestration (Docker, Kubernetes)
   # - Checks /actuator/health endpoint every 30 seconds
   # - Allows 40 seconds startup time before declaring unhealthy
   # - Considers container unhealthy after 3 failed checks
   HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \
       CMD wget --no-verbose --tries=1 --spider http://localhost:9000/actuator/health || exit 1

   # Expose port (documentation only; port is configurable via SERVER_PORT env var)
   EXPOSE 9000

   # Run application with environment variable support
   # - sh -c allows variable substitution for $JAVA_OPTS and other vars at runtime
   ENTRYPOINT ["sh", "-c", "exec java $JAVA_OPTS -jar app.jar"]
   ```

4. Save the file at project root
</action>

   ## Database Migrations

   ### Liquibase Auto-Migration

   Liquibase migrations run automatically on application startup:

   ```yaml
   spring:
     liquibase:
       enabled: true
       change-log: classpath:db/changelog/db.changelog-master.xml
   ```

   **Migration Process:**
   1. Application starts
   2. Readiness probe returns DOWN
   3. Liquibase applies pending changesets
   4. Schema is validated against Liquibase tracking table
   5. Readiness probe returns UP
   6. Application accepts traffic

   ### Zero-Downtime Deployments

   To enable zero-downtime deployments:

   1. **Make migrations backward-compatible:**
      - Add columns with DEFAULT values (don't drop)
      - Add new tables without constraints to existing data
      - Use Liquibase `createIndex` before schema changes

   2. **Use rolling updates:**
      - Deploy new version alongside old version
      - Route new requests to new version
      - Drain connections from old version
      - Old version exits gracefully

   3. **Kubernetes example:**
      ```yaml
      kind: Deployment
      spec:
        strategy:
          type: RollingUpdate
          rollingUpdate:
            maxSurge: 1
            maxUnavailable: 0
        template:
          spec:
            terminationGracePeriodSeconds: 30
            lifecycle:
              preStop:
                exec:
                  command: ["/bin/sh", "-c", "sleep 15"]
      ```

   ## Troubleshooting

   ### Container Won't Start

   **Check logs:**
   ```bash
   docker logs auth-service
   ```

   **Common issues:**
   - Database unreachable: Verify DB_URL, DB_USERNAME, DB_PASSWORD
   - Port in use: Try different port: `-p 9001:9000`
   - Out of memory: Increase Xmx in JAVA_OPTS

   ### Database Connection Fails

   ```bash
   # Test connectivity from container
   docker exec auth-service psql -h ${DB_URL} -U ${DB_USERNAME} -c "SELECT 1"
   ```

   **Solutions:**
   - Verify network connectivity between containers
   - Check database credentials
   - Ensure database service is healthy: `docker ps` and check postgres status
   - Check firewall rules (if using cloud provider)

   ### Migrations Won't Complete

   Check Liquibase logs:
   ```bash
   docker logs auth-service | grep liquibase
   ```

   **Common issues:**
   - Existing data incompatible with new schema: Check migration compatibility
   - Lock table held by another instance: Kill competing process
   - Migration SQL syntax error: Test against staging first

   ### Health Check Fails

   **Check health endpoint directly:**
   ```bash
   docker exec auth-service wget -q -O - http://localhost:9000/actuator/health
   ```

   **Common causes:**
   - Application still starting (check logs)
   - Database down (verify DB_URL)
   - Redis down (verify REDIS_HOST if configured)
   - Out of memory (increase -Xmx)

   ### Performance Issues

   **Monitor resource usage:**
   ```bash
   docker stats auth-service
   ```

   **Optimization:**
   - Increase connection pool: `DB_POOL_SIZE=30`
   - Increase JVM heap: `-Xmx2048m`
   - Enable Redis caching: `REDIS_HOST=...`
   - Review slow query logs

## Notes

**Dockerfile Design Decisions:**
- Multi-stage build separates compilation environment from runtime, reducing final image size
- Alpine base images (21-jdk-alpine for build, 21-jre-alpine for runtime) minimize image footprint
- Non-root user (appuser) enhances security by preventing privilege escalation exploits
- G1GC garbage collector tuned for low-latency, predictable pause times in containers
- HEALTHCHECK endpoint enables Docker and Kubernetes to monitor application health

**Docker Compose Design:**
- Three services (PostgreSQL, Redis, auth-service) with explicit healthchecks
- All configuration via environment variables (no hardcoded secrets)
- Service dependencies managed with `depends_on: condition: service_healthy`
- Persistent volumes for database and Redis data
- Bridge network isolates services from host and other applications

**Configuration Profiles:**
- `application-prod.yaml`: Production-hardened (INFO logging, Redis sessions, validate-only schema)
- `application-staging.yaml`: Production-like with test-friendly defaults (DEBUG logging, easier health checks)
- `application-integration-test.yaml`: (from Plan 1) Test-focused with H2 support
