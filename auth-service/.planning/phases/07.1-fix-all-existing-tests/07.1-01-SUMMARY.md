# Phase 7.1 Plan 1: Fix All Existing Tests Summary

All 96 tests pass with 0 errors and 0 failures after fixing five root causes across three categories.

## Root Causes and Fixes

### 1. Thymeleaf Null-to-Boolean Conversion (Rule 1 - Bug)

**Root Cause:** `th:if="${!emailMatch}"` in `invitation-landing.html` line 175 fails with `SpelEvaluationException: Type conversion problem, cannot convert from null to boolean` when `emailMatch` is not in the model (the old `InvitationController` renders this template without setting `emailMatch`).

**Fixes Applied:**
- Changed `th:if="${!emailMatch}"` to `th:unless="${emailMatch}"` — Thymeleaf's `th:unless` treats null as falsy without throwing
- Changed `th:if="${tokenInvalid || tokenExpired}"` to `th:if="${tokenInvalid == true || tokenExpired == true}"` — explicit boolean comparison handles null safely
- Added `model.addAttribute("emailMatch", true)` to `InvitationController.landingPage()` when an existing user is found

### 2. Missing `auth/invitation-error` Template (Rule 3 - Blocking)

**Root Cause:** `InvitationPageController` returns view name `"auth/invitation-error"` for invalid/expired tokens but this template did not exist, causing `TemplateInputException` at test runtime.

**Fix:** Created `src/main/resources/templates/auth/invitation-error.html` with standard layout, error code display, message, and support contact.

### 3. Missing DataSource URL for `@SpringBootTest` Tests (Rule 3 - Blocking)

**Root Cause:** `InvitationFlowIntegrationTest` and `EmailTemplateRenderingTest` used `@SpringBootTest` without `@ActiveProfiles`. The default `application.yaml` has `url: ${DB_URL}` which resolves to an empty string without an environment variable, causing `'url' must start with "jdbc"` at context initialization.

**Fix:** Added `@ActiveProfiles("local")` to both test classes. The `application-local.yaml` in test resources provides `jdbc:h2:mem:auth_test_${random.uuid}` with H2.

### 4. Security Configuration Missing Public Paths (Rule 2 - Missing Critical Functionality)

**Root Cause:** The security config did not include `/invitations/**` in `permitAll`, so the new `InvitationPageController` at `/invitations/{token}` redirected unauthenticated requests to login (302 instead of 200). Also, `/api/v1/invitations/*/status` was documented as "publicly accessible" in the controller but the security filter chain required JWT for all `/api/v1/**` endpoints, returning 401.

**Fixes:**
- Added `/invitations/**` to `defaultSecurityFilterChain` permitAll list
- Added explicit `permitAll()` for `/api/v1/invitations/*/status` in `managementApiSecurityFilterChain`

### 5. Missing `slug` Field in Test Fixture (Rule 1 - Bug)

**Root Cause:** `InvitationFlowIntegrationTest.setup()` created `Company.builder().name("Test Organization").build()` without setting the `slug` field, which is `@Column(nullable = false, unique = true)`. This caused `DataIntegrityViolationException` at test setup.

**Fix:** Added `.slug("test-organization-" + UUID.randomUUID().toString().substring(0, 8))` to the Company builder in `@BeforeEach`.
