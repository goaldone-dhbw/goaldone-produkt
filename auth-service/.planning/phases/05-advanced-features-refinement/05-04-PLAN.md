## Goal

**In Scope:**
- Invitation landing page (email detection, flow branching)

- Session management after acceptance

**Out of Scope:**
- Domain-based invitation eligibility (deferred to Phase 6)
- Batch invitation sending (out of scope)

- Analytics/reporting on invitation flows

### 4. Implement Invitation Status Query Endpoint

**New Endpoint:**
- `GET /api/v1/invitations/{token}/status`


- Used by frontend to decide whether to show acceptance form or status message

**Success Criteria:**

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Email matching false positive (shared family email) | Unintended account linking | Require authentication confirmation before linking; log all attempts; support account recovery |
| Token reuse after acceptance | Security vulnerability | Invalidate token immediately after use; track accepted state in database; validate expiration |
| Session management creates new session on linking | Login loop | Test session handoff carefully; use Spring Session JDBC established patterns |
| Performance: email lookup on every invitation viewed | Slow page load | Cache results with reasonable TTL; use indexed queries; consider rate limiting |
| Invitation expiration job runs slowly | Large cleanup time | Batch expire by chunks; use database bulk update; monitor execution time |
