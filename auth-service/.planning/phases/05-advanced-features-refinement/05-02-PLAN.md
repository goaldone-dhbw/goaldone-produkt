# Phase 5.2: Business Constraints (Last-Admin/Last-Super-Admin Checks)

**Status:** Ready for execution  
**Priority:** High (security-critical)  
**Estimated Effort:** 20-25 hours

---

## Goal

Implement validation logic to prevent organizational and system-level operational chaos by protecting the last administrator in an organization and the last system super-admin. Return RFC 7807 Problem Detail responses (409 Conflict) when constraints are violated.

---

## Scope

**In Scope:**
- Last-COMPANY_ADMIN check: prevent removing last admin from organization
- Last-COMPANY_ADMIN demote check: prevent changing last admin's role to USER
- Last-SUPER_ADMIN check: prevent removing super_admin flag from last system admin
- Service-layer validation in UserManagementService
- RFC 7807 Problem Detail error responses (409 Conflict)
- Error messages guide users to alternatives (promote another user, etc.)
- Audit logging for all constraint enforcement

**Out of Scope:**
- Role-based access control (existing ACL system out of scope)
- Organizational hierarchy beyond single org-user-role binding
- Bulk operation constraints (one-off for now; bulk operations out of scope)
- Custom status codes beyond 409

### 3. Add Last-Admin Check Methods to UserManagementService

**File:** `src/main/java/de/goaldone/authservice/service/UserManagementService.java`

**Changes:**
- Add method `isLastCompanyAdmin(Long userId, Long organizationId): boolean` that:
  - Queries Membership repository for users with role COMPANY_ADMIN in organization
  - Counts active memberships (status != DELETED)
  - Returns true if exactly one admin and that admin is userId
  - Uses CompanyRepository and MembershipRepository joins
- Add method `isLastSuperAdmin(Long userId): boolean` that:
  - Queries User repository for users with super_admin = true
  - Counts active super-admins
  - Returns true if exactly one super-admin and that admin is userId
- Both methods: log invocation with user context for audit trail

**Success Criteria:**
- Last-admin detection accurate across org boundaries
- Super-admin detection system-wide
- Query performance acceptable (use database-level COUNT if needed)
- Null-safe for users without memberships

## Acceptance Criteria

1. **Last-Admin Protection:** Organization with one admin → deletions/demotions blocked with 409
2. **Last-Super-Admin Protection:** System with one super-admin → status removal blocked with 409
3. **Error Format:** All constraint violations return RFC 7807 ProblemDetail with violationType field
4. **User Guidance:** Error detail includes actionable suggestion (promote another user, etc.)
5. **Audit Trail:** All constraint checks logged with user ID, organization, timestamp, violation type
6. **Non-Blocking Paths:** Legitimate deletions and role changes unaffected (promote to admin, create new admin)
7. **Transactional Safety:** Constraint check and mutation atomic (no partial states)

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Query performance for last-admin check on large orgs | Slow endpoints | Use database-level COUNT; add index on (organization_id, role, status) |
| Operator mistake: can't remove admin who left company | Support burden | Document workaround: promote a user, then remove original; consider "suspended" status |
| Last super-admin assumption: system requires at least one | Data corruption if false | Validate assumption in Phase 5 UAT; consider allowing zero super-admins in future |
| Concurrent modifications: race condition on last-admin count | Consistency error | Use pessimistic locking (SELECT FOR UPDATE) or optimistic versioning; document behavior |
| Integration with pending role-based ACL changes | Feature conflict | Coordinate with Phase 6 ACL team; current implementation is service-layer independent |
