# Phase 5.1: Account Linking Flow (F190) Integration

**Status:** Ready for execution  
**Priority:** High (blocks F190 completion)  
**Estimated Effort:** 40-50 hours

---

## Goal

Implement account linking functionality integrated into the invitation acceptance flow. Users invited to join an organization with a matching email address can link existing accounts instead of creating new ones. All linking requires OIDC-PKCE authentication.

---

## Scope

**In Scope:**
- Email matching logic (primary + secondary) during invitation acceptance
- UI flow for same-email and new-email scenarios on invitation landing page
- Invitation token acceptance endpoint integration (`POST /api/v1/invitations/{token}/accept`)
- OIDC-PKCE authentication flow for account verification
- Account consolidation (email addition, role assignment, no new user creation)
- Account linking confirmation email template

**Out of Scope:**
- Self-service account linking outside invitation flows
- Domain-based self-registration (deferred to Phase 6)
- Email address change/management features
- Advanced account merge conflict resolution beyond specified flow

## Acceptance Criteria

1. **Email Detection:** User invited with existing email → system detects match and presents linking option
2. **New Account Path:** User invited with new email → default create password flow works unchanged
3. **Linking Authentication:** User choosing to link → OIDC-PKCE flow completes with context preserved

5. **Confirmation:** Linking completes → confirmation email sent, user informed
6. **Audit Trail:** All linking attempts logged with user IDs, email, timestamp, organization

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Email matching collides with existing secondary emails | User confusion, unintended links | Validate email uniqueness during secondary email add; log all adds; require authentication for all linking |
| Linking context lost across OAuth redirect | Flow breaks, user stuck | Store in session + encrypted state; validate integrity on retrieve |
| Password reset for linked accounts broken | User support burden | Test secondary email in password reset flow; ensure reset emails sent to primary |
| Race condition: user deleted between match check and link | Consistency error | Use transaction scope for entire linking operation; catch user-not-found exception |
| Circular linking (A links to B, B links to A) | Account merge conflict | Validate linking is one-directional; authenticated user determines primary; prevent reverse linking |
