## What We Fixed

### Root Cause: stdin/stdout redirection conflict
When `exec >> "$LOG_FILE" 2>&1` redirects logging to a file, all subsequent `read` commands fail because stdin is no longer the terminal.

### Solutions Applied

1. **All `read` commands now use `/dev/tty`** (6 locations fixed)
   - Line 169, 177: `prompt_input` function
   - Line 546: recovery action choice
   - Line 779: UFW configuration
   - Line 821: clone repository choice
   - Line 926: .env file choice
   - Line 1176: terraform plan confirmation
   - Line 1300: reset-state confirmation

2. **Master key generation** - Replaced problematic `tr | head` pipe
   - Now uses `openssl rand -hex 16` (primary method)
   - Fallback to base64 + od if openssl unavailable
   - Avoids `set -o pipefail` issues

3. **HashiCorp repository setup** - Added retry logic
   - 3 attempts with exponential backoff (2s → 4s → 8s)
   - Graceful degradation if repo setup fails
   - Terraform becomes optional

4. **Environment template lookup** - Fixed path resolution
   - Now checks: `infra-setup/.env-example` → `.env-example` → `infra-setup/.env-dev-example`

5. **Comprehensive logging** - Added timestamps and error context throughout
   - Every critical step logged to deploy.log
   - Error capture from failed commands
   - Pre-prompt state verification
