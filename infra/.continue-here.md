## Key Decisions Made

1. **Dynamic user detection via $SUDO_USER** - Works for any deployment user (goaldone, ubuntu, root, etc.), not hardcoded
2. **Pause after docker group setup** - Allows time for group membership to activate (requires logout/login)
3. **Non-root resume from step 4+** - Docker operations don't need sudo once group is active
4. **Two-script architecture** - Much cleaner separation of concerns than conditional root checks

## Context

The deploy script is for GoalDone infrastructure setup. It orchestrates:
- Package installation (Docker, Terraform, Git, jq)
- Infrastructure provisioning (Zitadel + PostgreSQL via Docker Compose)
- Terraform initialization and deployment
- Token extraction and setup completion

The docker group issue was critical because:
- Docker operations need access to `/var/run/docker.sock` 
- User must be in `docker` group or have sudo privileges
- `sudo` creates a restricted environment that blocks docker discovery
- Solution: Separate sudo-required setup from user operations, require logout/login to activate group
