# Phase 07-01 Plan: API Regeneration & Verification

## Goal
Regenerate the TypeScript API client from the current openapi.yaml specification and systematically identify all API signature changes from Phase 5. This prerequisite establishes the authoritative API shapes for all downstream component updates and ensures no surprises during integration.

## Dependencies
- Phase 6.1 complete (backend and auth-service running)
- openapi.yaml finalized and validated

## Success Criteria
1. API client regenerates without errors: `npm run generate-api` completes successfully
2. All generated types in `frontend/src/app/api/` are valid TypeScript
3. Breaking changes documented:
   - Member operations: userId now UUID (not string)
   - MemberResponse shape: userId, accountId, email, firstName, lastName, role, status, createdAt
   - Any endpoint signature changes identified and listed
4. Component code audited against regenerated API shapes
5. Test fixtures updated to use new API shapes (preparation for Plan 07-02)

1. Open `frontend/src/app/api/model/memberResponse.ts` and document the shape:
   ```

   - email: string (required)
   - firstName?: string | null
   - lastName?: string | null

   - status: MemberStatus (required) — enum INVITED or ACTIVE
   ```

2. Grep for all usages of memberResponse types in components:
   ```bash

   ```
3. Document any files that reference members or user IDs
