  ```json
    { "id": "org-uuid-1", "slug": "acme", "role": "ROLE_ADMIN" },

  ]
  ```
- JWT `user_id` claim: unique user identifier (string)
- JWT `primary_email` claim: user's primary email

  <action>
Refactor the role extraction logic in AuthService to read from the flat `authorities` claim instead of Zitadel's nested URN structure. Make these specific changes:

1. **Replace getUserRoles() method:**
   - OLD: Searches for Zitadel URN keys like `urn:zitadel:iam:org:project:roles` and extracts object keys
   - NEW: Extract `authorities` claim (flat array of strings), map to org contexts
   - Return object structure: `{ [orgId: string]: string[] }`
   - Logic:
     1. Get decoded token
     2. Extract `authorities` array (default to [])
     3. Extract `orgs` array (default to [])
     4. For each org in orgs, find matching roles in authorities that belong to that org
     5. Return map: `{ org1_id: ['ROLE_ADMIN'], org2_id: ['ROLE_MEMBER'] }`
   
   Alternative simpler approach (if authorities are pre-scoped per org in JWT):
   - Just return the raw authorities array grouped by org from the `orgs` claim's role field
   - Example: orgs[0] has `role: "ROLE_ADMIN"` → authorities["org-uuid-1"] = ["ROLE_ADMIN"]

   Use the simpler approach: The `orgs` claim array already contains the role per org. Return a map built from orgs.

2. **Keep getUserMemberships() as-is:** It already returns the `orgs` array correctly.

3. **Add getOrganizations() method:**
   - Return the `orgs` array from JWT
   - Same as getUserMemberships() — can be an alias or keep separate for clarity
   - Return type: Array<{ id: string; slug: string; role: string }>

4. **Add getActiveOrganization() method:**
   - Return the first org in the `orgs` array (default active org)
   - Or null if user has no orgs (error state)
   - Used by dialogs and pages to initialize org context (D-12)

5. **Remove all Zitadel URN references:**
   - Delete the old getUserRoles() logic that searches for `urn:zitadel:iam:org:project:roles`
   - Keep getUserOrganizationId() for backward compatibility if needed, but mark as legacy

6. **Update method signatures and JSDoc:**
   ```typescript
   /**
    * Returns roles extracted from the 'authorities' claim, mapped by organization ID.
    * @returns Object where keys are org IDs and values are role arrays (e.g., { "org-uuid-1": ["ROLE_ADMIN"] })
    */
   getUserRoles(): { [orgId: string]: string[] }

   /**
    * Returns the user's organization memberships from the 'orgs' claim.
    * @returns Array of org objects with id, slug, and role
    */
   getOrganizations(): Array<{ id: string; slug: string; role: string }>

   /**
    * Returns the first organization (default active org after login).
    * @returns First org object or null if user has no orgs
    */
   getActiveOrganization(): { id: string; slug: string; role: string } | null
   ```

  <action>
Create a new service to manage organization context selection for dialogs and pages. This service centralizes the org selection logic and supports both dialog-scoped (D-09) and page-scoped (D-10) selections.

File: `frontend/src/app/core/services/org-context.service.ts`

Implementation requirements:

1. **Service class with Angular injection:**
   ```typescript
   @Injectable({ providedIn: 'root' })
   export class OrgContextService {
     private authService = inject(AuthService);
     // Dialog context: in-memory, cleared on dialog close
     private dialogOrgContext = new BehaviorSubject<string | null>(null);
     // Settings page context: persists for page session only
     private settingsOrgContext = new BehaviorSubject<string | null>(null);
   }
   ```

2. **Methods for dialog-scoped selection (D-09):**
   - `setDialogOrg(orgId: string)` — Set org for current dialog
   - `getDialogOrg()` — Get selected org (or null if not selected)
   - `clearDialogOrg()` — Clear selection when dialog closes
   - `dialogOrgContext$` — Observable for reactive UI binding

3. **Methods for page-scoped selection (D-10):**
   - `setSettingsOrg(orgId: string)` — Set org for settings page
   - `getSettingsOrg()` — Get selected org (or null)
   - `clearSettingsOrg()` — Clear when navigating away from settings
   - `settingsOrgContext$` — Observable for reactive UI binding

4. **Helper methods:**
   - `hasMultipleOrgs()` — Returns true if user is member of 2+ orgs (used to show/hide dropdowns)
   - `getOrganizations()` — Delegates to AuthService.getOrganizations()
   - `getDefaultOrg()` — Returns first org (from AuthService.getActiveOrganization())
   - `validateOrgAccess(orgId: string)` — Check if user is member of specified org

5. **Dialog context pattern (example usage):**
   ```
   // In Add Task Dialog:
   constructor(private orgContext: OrgContextService) {}
   ngOnInit() {
     this.orgContext.clearDialogOrg(); // Fresh start
     if (this.orgContext.hasMultipleOrgs()) {
       // Show dropdown
     }
   }
   onOrgSelected(orgId: string) {
     this.orgContext.setDialogOrg(orgId);
   }
   onSubmit() {
     const orgId = this.orgContext.getDialogOrg() || this.orgContext.getDefaultOrg()?.id;
     // POST /api/v1/tasks with X-Org-ID header
   }
   ngOnDestroy() {
     this.orgContext.clearDialogOrg();
   }
   ```

6. **Settings page pattern (example usage):**
   ```
   // In Company Settings:
   constructor(private orgContext: OrgContextService) {}
   ngOnInit() {
     if (this.orgContext.hasMultipleOrgs()) {
       // Show dropdown, default to first org
       this.orgContext.setSettingsOrg(this.orgContext.getDefaultOrg()?.id || '');
     }
   }
   onOrgSelected(orgId: string) {
     this.orgContext.setSettingsOrg(orgId);
   }
   ngOnDestroy() {
     this.orgContext.clearSettingsOrg();
   }
   ```

Per D-09: Dialog-Scoped Org Selection.
