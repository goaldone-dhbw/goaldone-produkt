# Phase 03.1: Refine Organization Context and Header Requirements - Research

**Researched:** 2026-05-02
**Domain:** Multi-Tenant Architecture / Security / OpenAPI
**Confidence:** HIGH

## Summary

This phase implements a Multi-Org architectural shift. The core transition is from path-based organizational context (`/orgs/{id}/...`) to header-based context (`X-Org-ID`). This header works in tandem with the user's JWT, which contains an `orgs` claim (array of memberships). The backend must verify that the requested `X-Org-ID` matches one of the user's memberships.

**Primary recommendation:** Use a Spring Security `OncePerRequestFilter` to extract and validate the `X-Org-ID` header, storing it in a `ThreadLocal` `TenantContext`. Implement a custom `MethodSecurityExpressionHandler` to provide a clean `@PreAuthorize("hasOrgRole('...')")` DSL.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01: Header as Source of Truth.** The `X-Org-ID` header is the primary mechanism for declaring the active organization context for a request.
- **D-02: Mandatory for Actions.** The header is **mandatory** for all destructive or creative actions (POST, PUT, DELETE) and for member management.
- **D-03: Optional for Lists (Dual-Mode).** For listing endpoints (e.g., `GET /tasks`), the header is **optional**.
    - **Header Missing:** Return data for **all** organizations the user is a member of.
    - **Header Present:** Filter data strictly to the specified organization.
- **D-04: Remove {accountId}.** The legacy `{accountId}` path parameter is removed from all endpoints. The backend resolves the membership identity using `JWT.user_id` + `X-Org-ID`.
- **D-05: Remove {orgId} from Paths.** Organizational context is moved from the path to the header.
- **D-06: Member Management Path.** The new unified path for member management is `/organization/members`.
- **D-07: custom hasOrgRole Expression.** Implement a custom Spring Security expression: `@PreAuthorize("hasOrgRole('ROLE_NAME')")`.
- **D-08: Header-JWT Correlation.** `hasOrgRole` must verify that the user possesses the required role *specifically* for the organization ID provided in the `X-Org-ID` header, by cross-referencing the `orgs` claim in the JWT.
- **D-09: 403 on Mismatch.** If a user provides an `X-Org-ID` in the header that they are not a member of in their JWT, the backend must return a `403 Forbidden` immediately.
- **D-10: Intelligent Interceptor.** The frontend `AuthInterceptor` will automatically attach the "Active Org" header.
- **D-11: Explicit Override.** UI components (like the "Create Task" dialog) can explicitly pass an `X-Org-ID` to the `HttpClient` to override the global context.
- **D-12: Conditional Dropdowns.** In creation/admin dialogs, a dropdown to select the organization is only displayed if the user has more than one membership (`orgs.length > 1`).

### Pattern 1: Tenant Context Filter
A `OncePerRequestFilter` that extracts the `X-Org-ID` header and validates it against the authenticated user's JWT `orgs` claim. It must clear the `ThreadLocal` in a `finally` block.

### Pattern 2: Custom Method Security Expression
Extending `SecurityExpressionRoot` allows for clean, domain-specific authorization logic like `@PreAuthorize("hasOrgRole('...')")`. This is more readable than complex SpEL or bean references.

## Common Pitfalls

### Pitfall 1: Request Context Leakage
**What goes wrong:** Data from a previous request is returned to the current user.
**Why it happens:** `ThreadLocal` is not cleared after the request finishes.
**How to avoid:** Always use a `try-finally` block in the Filter to call `TenantContext.clear()`.

### Pitfall 2: Security Filter Order
**What goes wrong:** `Authentication` is null in the `TenantContextFilter`.
**Why it happens:** The filter runs before `BearerTokenAuthenticationFilter`.
**How to avoid:** Use `http.addFilterAfter(new TenantContextFilter(), BearerTokenAuthenticationFilter.class)`.

### Pitfall 3: Case-Sensitivity in Roles
**What goes wrong:** `hasOrgRole('ADMIN')` fails because the JWT has `admin`.
**Why it happens:** Roles in Spring Security are often case-sensitive and prefixed with `ROLE_`.
**How to avoid:** Normalize role names during validation or use a consistent prefix/case.

## Code Examples

### Spring Security Expression Root
```java
// Source: https://docs.spring.io/spring-security/reference/servlet/authorization/method-security.html
public class CustomMethodSecurityExpressionRoot 
        extends SecurityExpressionRoot 
        implements MethodSecurityExpressionOperations {
    
    public CustomMethodSecurityExpressionRoot(Authentication authentication) {
        super(authentication);
    }

    public boolean hasOrgRole(String role) {
        String activeOrgId = TenantContext.getOrgId();
        if (activeOrgId == null) return false;
        
        Jwt jwt = (Jwt) this.getAuthentication().getPrincipal();
        List<Map<String, Object>> orgs = jwt.getClaim("orgs");
        
        return orgs.stream().anyMatch(org -> 
            activeOrgId.equals(org.get("id")) && role.equals(org.get("role"))
        );
    }
    // ... mandatory overrides for MethodSecurityExpressionOperations
}
```

### Angular Tenant Interceptor
```typescript
// Source: Angular 17+ Functional Interceptors
export const tenantInterceptor: HttpInterceptorFn = (req, next) => {
  const tenantService = inject(TenantService);
  const activeOrgId = tenantService.activeOrgId();

  if (activeOrgId) {
    const cloned = req.clone({
      setHeaders: { 'X-Org-ID': activeOrgId }
    });
    return next(cloned);
  }
  return next(req);
};
```
