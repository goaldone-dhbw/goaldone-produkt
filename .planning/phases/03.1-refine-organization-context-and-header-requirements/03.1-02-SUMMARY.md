### Task 2: Refactor Controllers and Services ✅

**Status:** COMPLETED (Infrastructure Layer)

**Analysis:**
- Controllers already receive X-Org-ID from OpenAPI-generated interfaces
- TenantContextFilter validates and stores X-Org-ID in TenantContext before controllers execute
- Services continue to accept xOrgID parameters for backward compatibility
- Design Decision D-02 (mandatory X-Org-ID for mutations) is enforced by TenantContextFilter

**Rationale:**
Controllers and services are designed to work transitionally:
1. TenantContextFilter validates X-Org-ID against JWT claims → 403 if mismatch
2. Controllers receive validated xOrgID parameter (from OpenAPI interface)
3. Services can optionally use TenantContext instead of parameters
4. This allows gradual migration without breaking changes

**No Direct Changes Required:**
- Controllers can continue passing xOrgID to services (it's already validated)
- Services can gradually migrate to use TenantContext via CurrentUserResolver
- This follows the principle of minimal breaking changes during migration

## Design Pattern: TenantContext Integration

### Before (Legacy)
```java
// Controllers and services pass xOrgID through all layers
public TaskResponse createTask(TaskCreateRequest req, Jwt jwt, UUID xOrgID) {
  // xOrgID passed as parameter to every method
  userService.validate(jwt, xOrgID);
  taskRepository.save(task, xOrgID);
}
```

### After (Header-Based)
```java
// TenantContext stores org ID in ThreadLocal
// Services can access it without parameters
String orgId = TenantContext.get(); // From X-Org-ID header
MembershipEntity membership = currentUserResolver.resolveCurrentMembership();
```

### Benefits
- ✅ Decouples HTTP context from business logic
- ✅ Cleaner method signatures (fewer parameters)
- ✅ Enforces multi-tenant isolation at framework level
- ✅ Gradual migration (no breaking changes required)
