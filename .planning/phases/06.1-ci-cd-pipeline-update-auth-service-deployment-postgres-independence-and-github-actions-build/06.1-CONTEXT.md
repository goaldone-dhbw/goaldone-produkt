# Phase 6.1: CI/CD Pipeline Update — Auth-Service Deployment, Postgres Independence, GitHub Actions Build

**Gathered:** 2026-05-03
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase updates the CI/CD infrastructure and deployment strategy to support the custom auth-service as a first-class service alongside the existing backend and frontend. Phase 6.1 executes AFTER Phase 6 completes (when backend compiles and tests pass).

Core deliverables:

1. **GitHub Actions Workflow Updates**
   - Add CI job to build auth-service Docker image and push to GHCR (alongside backend/frontend)
   - Update version tagging and .env template to include auth-service versioning

2. **Dockerfiles & Docker Compose**
   - Create Dockerfile for auth-service (separate image, not embedded in backend)
   - Update docker-compose files (dev/prod) to deploy auth-service as a standalone container
   - Create separate PostgreSQL instances for auth-service and backend
   - Define startup order via docker-compose depends_on

3. **Deployment Script Updates**
   - Expand `deploy.sh` to orchestrate auth-service startup alongside backend/frontend
   - Add health checks for auth-service JWKS endpoint validation
   - Implement rollback support for auth-service via version management

4. **Configuration Management**
   - Add AUTH_SERVICE_* environment variables to .env templates (dev/prod)
   - Document how RSA keys are injected at deployment time via GitHub Secrets
   - Update frontend env.js injection to include auth-service issuer URI and client ID

5. **Infrastructure Cleanup**
   - Remove old Zitadel infrastructure (infra/infra-setup/) — complete cutover to auth-service

The result: Auth-service and backend operate as independent, scalable services with separate databases, versioned deployments, and automated health checks.

</domain>

<decisions>
## Implementation Decisions

### Deployment Architecture

- **D-01: Separate Docker Container.** Auth-service runs in its own Docker container with a distinct image (separate from backend). Provides isolation, independent lifecycle management, and future scaling flexibility.

- **D-02: Separate PostgreSQL Instances.** Auth-service has its own PostgreSQL database instance (separate from backend). Backend and auth-service do not share schemas or databases. Eliminates coupling and enables independent scaling/migration.

- **D-03: Explicit Startup Order.** Docker-compose defines startup order via `depends_on`:
  1. `postgres-auth` (auth-service database)
  2. `postgres-backend` (backend database)  
  3. `auth-service` (depends on postgres-auth)
  4. `backend` (depends on postgres-backend AND auth-service — must wait for JWKS endpoint to be live)
  5. `frontend` (depends on backend)

  This ensures auth-service is available before backend attempts to use its JWKS endpoint for token validation.

- **D-04: GitHub Actions Separate Build Job.** Add a dedicated CI job to build and push auth-service Docker image on every master push (alongside backend/frontend). Auth-service gets versioned tags matching the release version (e.g., `v0.1.0`).

### Key Management & Secrets

- **D-05: GitHub Secrets → Runtime Injection.** Auth-service RSA private key (used for JWT signing) is stored in GitHub Secrets and injected into the `.env` file at deployment time by the CD pipeline. Keys are **regenerated on each deployment**, replacing the previous key. This ensures key material is never committed to the repository.

  **Important:** This differs from Phase 1's requirement for RSA key persistence across restarts. The solution here is volume-based persistence on the VPS: the key lives in `/path/to/auth-service/keys/` (mounted via docker volume). The CD pipeline injects the secret value into that location at first deployment; subsequent restarts read from the volume.

### Deployment & Health Checks

- **D-06: Expand deploy.sh to Manage Auth-Service.** The existing `deploy.sh` script (currently managing Postgres → Backend → Frontend) is extended to include auth-service orchestration:
  1. Ensure `postgres-auth` is healthy
  2. Start `auth-service` and wait for health check
  3. Verify JWKS endpoint is reachable (`GET /oauth2/jwks`)
  4. Only then proceed to Postgres → Backend → Frontend startup
  5. Implement version management for auth-service (like backend/frontend)

- **D-07: JWKS Endpoint Health Check.** Before backend is allowed to start, deploy script validates auth-service liveness by fetching the JWKS endpoint (`http://auth-service:8080/oauth2/jwks`). This ensures the auth-service token validation infrastructure is ready.

- **D-08: Rollback Support for Auth-Service.** Auth-service versions are managed in `.env` (e.g., `AUTH_SERVICE_VERSION=0.1.0`). If auth-service deployment fails health checks, the script exits. Ops can manually roll back by setting `AUTH_SERVICE_VERSION` to a known-good previous version and re-running deploy.sh. This matches the rollback strategy for backend/frontend.

### Environment Configuration

- **D-09: AUTH_SERVICE_* Variables in .env Templates.** Both `docker/dev/.env.example` and `docker/prod/.env.example` include:
  - `AUTH_SERVICE_VERSION` — Docker image tag (same format as BACKEND_VERSION)
  - `AUTH_SERVICE_DB_HOST` — Postgres hostname for auth-service (e.g., `postgres-auth-dev`)
  - `AUTH_SERVICE_DB_PORT` — Database port (5432)
  - `AUTH_SERVICE_DB_NAME` — Database name (e.g., `auth_service`)
  - `AUTH_SERVICE_DB_USER` — Database user
  - `AUTH_SERVICE_DB_PASSWORD` — Database password (stored in GitHub Secrets)
  - `AUTH_SERVICE_RSA_PRIVATE_KEY` — RSA private key (injected from GitHub Secrets at runtime)
  - `AUTH_SERVICE_JWT_ISSUER_URI` — Issuer claim in JWT (e.g., `https://auth.goaldone.de`)

- **D-10: Frontend OIDC Config via Runtime Injection.** Frontend's `env.js` continues the dynamic configuration pattern (used for Zitadel):
  - `OIDC_ISSUER_URI` → points to auth-service issuer URI
  - `OIDC_CLIENT_ID` → frontend client ID registered in auth-service
  
  These are injected at deployment time by the CD pipeline into the frontend `.env.js` file before the frontend container starts. Same approach as before, just different issuer.

- **D-11: Backend Spring Profile Configuration.** Backend continues to use Spring profiles (local, dev, prod). Auth-service configuration in `application-dev.yaml` / `application-prod.yaml` specifies:
  - `spring.security.oauth2.resourceserver.jwt.issuer-uri` → auth-service issuer
  - `spring.security.oauth2.resourceserver.jwt.jwks-uri` → auth-service JWKS endpoint (derived from issuer URI)
  
  No hardcoded URLs; all driven by issuer-uri (which reads from environment).

### CI/CD Updates

- **D-12: Version Tagging Unification.** CD pipeline tags releases with a single semantic version (e.g., `v0.2.0`) and applies it to all three services:
  - `ghcr.io/goaldone-dhbw/goaldone-auth-service:0.2.0`
  - `ghcr.io/goaldone-dhbw/goaldone-backend:0.2.0`
  - `ghcr.io/goaldone-dhbw/goaldone-frontend:0.2.0`

  All services deployed with matching version numbers for consistency.

- **D-13: Always Build Auth-Service.** Auth-service Docker image is built on every master push (no path-based conditional build). Simpler CI logic, ensures consistency.

### Infrastructure & Cleanup

- **D-14: Remove Zitadel Infrastructure.** Delete `infra/infra-setup/` and all Zitadel-related docker-compose files and documentation. Complete cutover to auth-service — no dual-IdP period or fallback to Zitadel.

- **D-15: New Auth-Service Deployment.** Auth-service is deployed via the same docker-compose + deploy.sh mechanism as backend/frontend. No separate Kubernetes cluster, no additional tools. Auth-service scales alongside the existing infrastructure.

### Scope Boundaries

- **D-16: Phase 6.1 Does Not Include Auth-Service Code Development.** This phase updates infrastructure ONLY. Auth-service code (API endpoints, database schema, M2M client flow) is complete from Phase 1. Phase 6.1 focuses on containerization, orchestration, and CI/CD.

- **D-17: No Multi-Environment Rollout Strategy.** Phase 6.1 uses simple blue-green for frontend and rolling restart for backend/auth-service. Canary deployments or advanced traffic splitting deferred to future phases.

</decisions>

<specifics>
## Specific Ideas

- **Auth-Service Container Name Convention:** Use `auth-service-dev` / `auth-service-prod` to match backend-dev/backend-prod naming. Consistent with existing patterns.

- **JWKS Endpoint Validation:** In deploy.sh, check JWKS endpoint with:
  ```bash
  curl -sf http://auth-service-${STAGE}:8080/oauth2/jwks > /dev/null || exit 1
  ```
  Must succeed before proceeding to backend startup. Ensures token validation will work.

- **Postgres Service Dependencies:** In docker-compose, use explicit health checks + depends_on conditions:
  ```yaml
  postgres-auth:
    healthcheck: ...
  auth-service:
    depends_on:
      postgres-auth:
        condition: service_healthy
  ```
  Don't rely on just `depends_on` without health checks — can lead to race conditions.

- **RSA Key Storage:** Auth-service Dockerfile should expect the key at a mounted path (e.g., `/secrets/auth-service-rsa-private.key`). CD pipeline writes the key to this path at deployment time. Volume persists across container restarts.

- **Frontend env.js Templating:** Current approach injects environment variables into `frontend/src/assets/env.js` at deployment time. Extend this to include auth-service issuer URI. Template something like:
  ```javascript
  window.__env = {
    OIDC_ISSUER_URI: '${AUTH_SERVICE_ISSUER_URI}',
    OIDC_CLIENT_ID: '${AUTH_SERVICE_FRONTEND_CLIENT_ID}',
    API_BASE_PATH: '${API_BASE_PATH}'
  };
  ```

- **GitHub Secrets Naming:** Use GitHub Secrets like:
  - `DEV_AUTH_SERVICE_RSA_PRIVATE_KEY`
  - `PROD_AUTH_SERVICE_RSA_PRIVATE_KEY`
  - `DEV_AUTH_SERVICE_DB_PASSWORD`
  - `PROD_AUTH_SERVICE_DB_PASSWORD`
  - `AUTH_SERVICE_ISSUER_URI_DEV` / `AUTH_SERVICE_ISSUER_URI_PROD`

- **Commit Message Template:** Use commits like:
  - `ci(06.1): add auth-service Docker image build job`
  - `ci(06.1): expand deploy.sh to orchestrate auth-service`
  - `docs(06.1): update docker-compose files for auth-service deployment`

</specifics>
