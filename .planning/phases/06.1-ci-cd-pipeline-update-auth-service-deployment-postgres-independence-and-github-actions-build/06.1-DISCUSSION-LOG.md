## Deployment Model

| Option | Description | Selected |
|--------|-------------|----------|
| Separate container | Auth-service runs in its own Docker container (separate image). Provides isolation and independent scaling, but more complex orchestration. | ✓ |
| Embedded in backend | Auth-service runs in the same Spring Boot process as the backend. Simpler deployment, shared health checks, but tighter coupling. | |
| Undecided | Let's clarify other decisions first, then come back to this. | |

**User's choice:** Separate container
**Notes:** Auth-service needs to be independently versioned, deployed, and scaled. Embedding would complicate the backend image and make auth-service updates risky.

---

## Database Isolation

| Option | Description | Selected |
|--------|-------------|----------|
| Shared database (one Postgres, separate schemas) | Both services read/write the same Postgres instance; separate schemas for isolation. Simpler ops, single backup/restore target. | |
| Separate databases (one Postgres, multiple databases) | Both services use the same Postgres instance but different databases. Cleaner logical isolation than schemas. | |
| Separate Postgres instances | Auth-service has its own Postgres. Maximum isolation but increases operational complexity and resource usage. | ✓ |

**User's choice:** Separate Postgres instances
**Notes:** Eliminates any coupling between backend and auth-service data models. Allows independent scaling and migration if needed in the future.

---

## Key Management

| Option | Description | Selected |
|--------|-------------|----------|
| GitHub Secrets → .env at runtime | Store RSA private key in GitHub Secrets; CD pipeline injects into .env on VPS. Keys rotate on redeployment. | ✓ |
| Persistent file on VPS | Generate key once during first deployment; store on VPS filesystem via Docker volume. Keys survive container restarts (required for token validation). | |
| Generated in auth-service startup | Auth-service generates new RSA key on startup if one doesn't exist (uses persistent volume). Simplest but requires strong volume management. | |

**User's choice:** GitHub Secrets → .env at runtime
**Notes:** Keys are injected at deployment time but persisted on the VPS via Docker volumes so they survive container restarts. This satisfies Phase 1's requirement that keys persist across restarts while keeping secrets out of version control.

## Deploy Script Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Expand deploy.sh to manage auth-service | Deploy script handles auth-service startup, health checks, and order-of-operations (Postgres → Auth-Service → Backend → Frontend). | ✓ |
| Keep deploy.sh for backend/frontend only | Auth-service is started/stopped via manual docker-compose commands or a separate deployment tool. Deploy.sh stays focused on backend/frontend. | |

**User's choice:** Expand deploy.sh to manage auth-service
**Notes:** Centralizes all deployment logic in one script. Simplifies ops — one command deploys the entire stack.

## Auth-Service Build

| Option | Description | Selected |
|--------|-------------|----------|
| Separate CI job for auth-service | GitHub Actions builds auth-service image and pushes to GHCR alongside backend/frontend. Requires separate Dockerfile and image tags. | ✓ |
| Build as part of backend | Auth-service is built as part of the backend Maven build (JAR). Same image, separate application within. | |

**User's choice:** Separate CI job for auth-service
**Notes:** Auth-service gets its own Docker image and GHCR repository entry, matching the backend/frontend pattern.

## Service Startup Order

| Option | Description | Selected |
|--------|-------------|----------|
| Postgres(auth) → Auth-Service → Postgres(backend) → Backend → Frontend | Auth-service starts first because backend depends on it for token validation. Requires two separate Postgres containers. | ✓ |
| Postgres(auth) → Postgres(backend) → Auth-Service → Backend → Frontend | Both Postgres instances start, then auth-service, then backend. More parallel startup. | |
| depends_on handles it automatically | Just define depends_on relationships; docker-compose orchestrates the startup order. Simplest in compose, but requires clear dependencies. | |

**User's choice:** Postgres(auth) → Postgres(backend) → Auth-Service → Backend → Frontend
**Notes:** Clear, explicit order. Auth-service is available before backend attempts to fetch the JWKS endpoint.

## Frontend OIDC Configuration

| Option | Description | Selected |
|--------|-------------|----------|
| Injected via env.js at runtime | Frontend's env.js dynamically loads OIDC config pointing to auth-service. Requires CI/CD to template env.js. | ✓ |
| Built into frontend image per environment | Dev frontend image has dev auth-service URI; prod image has prod auth-service URI. Baked into build, no runtime injection needed. | |

**User's choice:** Injected via env.js at runtime
**Notes:** Maintains the existing pattern used for Zitadel. No need to rebuild frontend images to swap environments.

## Legacy Cleanup

| Option | Description | Selected |
|--------|-------------|----------|
| Keep for reference/rollback | Keep the Zitadel docker-compose in the repo as a fallback. Document it as 'legacy' so new team members don't accidentally use it. | |
| Remove completely | Delete Zitadel docker-compose and docs. Clean break from the old system once auth-service is proven in production. | ✓ |

**Notes:** Complete cutover to auth-service. No fallback to Zitadel.

## Pre-Deployment Validation

| Option | Description | Selected |
|--------|-------------|----------|
| Health check auth-service, verify JWKS endpoint | Deploy script hits auth-service /oauth2/jwks endpoint to confirm it's operational before starting backend. Prevents bad deployments. | ✓ |
| Simple HTTP health check | Deploy script does a simple HTTP check (e.g., /health or /actuator/health) to auth-service. Lighter validation. | |
| Assume docker-compose depends_on is sufficient | Trust docker-compose's service ordering. No extra validation needed. | |

**User's choice:** Health check auth-service, verify JWKS endpoint
**Notes:** JWKS endpoint validation ensures the backend will be able to validate JWT tokens upon startup.

## Configuration Template

| Option | Description | Selected |
|--------|-------------|----------|
| Add AUTH_SERVICE_* variables to .env templates | Update docker/dev/.env.example and docker/prod/.env.example with auth-service config (DB URL, JWT issuer URI, etc.). | ✓ |
| Separate auth-service/.env.example | Auth-service has its own .env.example in a dedicated directory, keeps concerns separate. | |

**User's choice:** Add AUTH_SERVICE_* variables to .env templates
**Notes:** Consistent with existing pattern. Makes it clear what configuration is needed.

## CI Trigger Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Always build auth-service on master push | Simplest: auth-service always rebuilt with backend/frontend. Ensures consistency, minimal configuration. | ✓ |
| Only build if auth-service/* changes | Use path filters to skip auth-service build if only backend/frontend changed. Faster CI for most commits, requires path-based workflow logic. | |
| Decide later | Not critical now; can be decided when writing the CD pipeline updates. | |

**User's choice:** Always build auth-service on master push
**Notes:** Simpler CI logic. All three services always tagged with the same version.

---

## Failure Recovery

| Option | Description | Selected |
|--------|-------------|----------|
| Support rollback to previous auth-service version | Like backend/frontend, auth-service should be versioned in .env and support rolling back on failure. | ✓ |
| Keep last 2 versions (canary/parallel) | New auth-service starts alongside old; if new fails health checks, traffic stays on old version. | |
| Manual intervention required | If auth-service fails, deployment stops. Ops intervenes manually. Simpler initial implementation. | |

**User's choice:** Support rollback to previous auth-service version
**Notes:** Matches the deployment pattern for backend/frontend. Versions are managed in .env; rollback is a simple env edit + redeploy.

## Summary

All gray areas resolved. Decisions captured in CONTEXT.md for downstream research and planning agents.

Key outcomes:
- Auth-service as separate, independently-versioned Docker container
- Separate PostgreSQL instances for auth-service and backend
- Secrets injected at runtime; keys persisted on VPS
- Expanded deploy.sh orchestrating all services
- JWKS endpoint validation before backend startup
- Complete removal of Zitadel infrastructure
